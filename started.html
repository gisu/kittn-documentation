Introduction

"Kittn" is a toolkit designed to support mainly the construction of frontends. Anyone looking for a colorful mixture of prefabricated elements, as one would expect from bootstrap and co, will not be happy with Kittn. Here rather the methods are offered to those elements on a simple way to generate and expand.

Around the CSS part, there are still many automatic functions installed that help the developer. For example, to generate image assets and distribute, to build HTML mockups, to make sprites to generate fallbacks for SVG files. That's why I also avoid the term framework.

Against the terminal you should have no fear :p

Dependencies

In order to work with "Kittn", you need to install some extensions and environments. Anyone who has installed these tools, which can skip the step.

Environments

The toolkit has been tested so far on Mac systems! On Windows systems, there can be problems.

  • Ruby >= 1.9.0
  • Node >= 0.10.10

On Mac systems, Ruby is already installed, this step would be omitted. But if you like you can update on Ruby v.2.12.

To install Ruby on Windows, simply use the One Click Installer.

Currently Sass is compiled with Ruby-Sass. When Libsass is 100% productive usable, the Ruby dependency is omitted.

Node can be installed on Mac systems either Brew or you can use the Installer, then for both systems.

Install Kittn

Once the dependencies are installed, the generator is installed.

npm install -g generator-kittn

The generator will automatically install Yeoman, Grunt and Bower, when they not installed on the system. Kittn is now available as a globally installed package.

To use Kittn, you created a project directory itself and activates the generator:

yo kittn

The generator will now ask a series of questions with which the package.json and bower.json` be configured.

Of greater importance are the last three questions:

CSS Filename : Determines the general file name of the CSS file.

Do you need IE8 compatibility? : If this question is answered with y, two CSS files are generated. In HTML files, calling the two files is then also available (otherwise only one CSS file is called).

Do you want to use Jade as HTML Compiler? : If this question is answered with y the Jade compiler is running permanently. When n, Jade is used only for the first compilation of the HTML files and then shut off.

After answering the last question, Yeoman will download all the project-related dependencies with Bower and NPM.

Build HTML with Jade

Jade is a HTML compiler that is in principle used for the first generation of the HTML files, then it is the developer free to decide for themselves whether they want to use Jade on. For example, who works directly with WordPress will not need Jade.

The Jade Files can also be found under src/:

src/
.. jade/ Jade files
.... components/ Jade componenten
.... mixins/ Jade Mixins
.... templatepart/ Template segments, eg HTML Header
.... templates/ Different template types

The Jade part is more like a little boilerplate. It's up to you how much you want to segment the Jade code.

Working with JS Files

Internal Files

Internal JS files are in the directory src/js/ saved, there will be discover app.js file - regularly it can be considered sufficient one file.

The app.js will load in HTML always at the end.

Once the default Gulp task is active, all files will be monitored in src/js/, at a change through this one JSHint test and are stored in dist/assets/js/ (with the same filename).

External Files

Let us turn to the external script files, this can be either Bower load in the package, if they are not found there, one can integrate over the scripts folder.

If you want to, for example, Add an additional scripts, you will first search for it - sometimes the packages are stored under a different name.

bower search package-name

If you have the appropriate package found, it is installed with the following command:

bower install package-name --save

Das entsprechende Paket wird in src/stash/bower/ installiert und zusätzlich in der bower.json notiert. Möchte man ein installiertes Package wieder deinstallieren:

The appropriate package is installed in src/stash/bower/ and also listed in the bower.json. If you want to uninstall an installed package again:

bower uninstall package-name --save

The contents of the bower Folders is not transferred to the external Git repo. Developers who work on the project, download the Dependencies Bower over bower install. Therefore, it is important to note each dependency in the bower.json.

Of course it can happen that you can not find certain packages via Bower, you can register then either yourself:

bower register https://url.to-the.git.repo package-name

Or you can load the package in src/stash/scripts/ where it is available for further use.

Extend Kittn

Kittn is designed to give the developer a skeleton in his hand, with which he can already work. Nevertheless, it will happen that not enough a the default plugins. One or the other would take the native js might use CoffeeScript might be missing that some other feature.

To expand Kittn you should list the new plugins on the package.json, as the node_modules directory is not transmitted to the git repo.

npm install packagename --save-dev

This is a package about NPM Network, installed into the project directory and listed in the package.json. Thus, the subsequent Developer installed exactly the packages that are needed for the project. It understood then you then the gulpfile.js and / or gruntfile.js must modify file.

To uninstall packages:

npm uninstall packagename --save-dev

Image preparing

The directory structure is particularly important for the preparation and processing of the images files.

In the directory src/stash/images/ there are two subfolders css-images/ and html-images - for the record differs "Kittn" between CSS and HTML embedded images. Within this folder, there are empty PSD files, they own but only for users who already work with Photoshop or Photoshop CC CC2014, because here is only the generator.

  • src/stash/images/css-images/sprite.psd Asset file for all sprite images (retina & nonRetina)
  • src/stash/images/css-images/single.psd Asset file for all single images
  • src/stash/images/css-images/texture.psd Asset file for all texture images
  • src/stash/images/html-images/html.psd Asset file for HTML Images

The syntax is relatively easy to handle:

  • filename.png Generates a PNG
  • 200% filename.gif Generates a gif with a 200% scaling
  • filename.jpg70% Generates a JPG with 70% quality
  • 200% filename@2x.png, filename.png Generates two PNG files, one in Double resolution (retina)

Please note the sprites file name of the retina Qualifying ends with @ 2x in NonRetina eliminates the postfix.

More information to the generator (German) - also how you can create with Photoshop SVG.

The generator will then generate new assets if something in the PSD file changes (eg move a Layer), but you do not even save the file. Once you gulp extended can run on the possibly be disturbing, because the processes are then started as well.

The PSDs are all already connected to the generator. Once you labeled Layer/ groups according to the generator syntax and stores, the assets are stored in their respective folders (with the suffix assets).

Work with the generator is optional if one should be too cumbersome to work or if you have no Photoshop CC at your disposal, which can also its assets as he is used to generate. The important thing is that always end up the image files in the appropriate directories.

It is very important to make sure that the PSDs (or any other File) can not be more than 100 MB in size, so before committing the file size should be controlled. Github responding when you push very 'allergic' to files over 100 MB, which often leads to termination.
