Setup

After the installation is complete, the project must be initialized.


gulp init

Init Process:

  • Build a individual Modernizr Build and Copy to dist
  • Copy System Files to dist
  • Copy JS Files to dist
  • Copy HTML Images from src/stash/images/html-images/html-assets/ to dist/images
  • Start the grunt Build Image Task
    • Copy to dist/assets/img/sprite/
    • Copy to dist/assets/img/sprite_2x/
    • Build Retina Sprite src/sass/maps/
    • Build NonRetina Sprite src/sass/maps/
    • Rebuild Retina Sprite Map dist/assets/img/
    • Rebuild NonRetina Sprite Map dist/assets/img/
    • SVG Minify and Copy to dist/assets/img/svg/
    • Build PNG Fallback Image dist/assets/img/svg
    • SVG Minify and Copy to dist/assets/img/svgonly/
    • Copy Texture Image to dist/assets/img/texture/
    • Copy Single Images to dist/assets/img/single/
    • Build Image Map src/sass/maps/
  • Compile Sass Files from src/sass/ into dist/assets/css/
  • Compile Jade Files from src/jade/ into dist/
  • Check JS File from src/js/ and copy it to dist/assets/js/
  • Combine some JS Files and copy it to dist/assets/js/scripts.js

When initializing, different files from scr directory will be copied to the distribution dist directory. In addition, js files are combined, html and css would be compiled.

Important! You will always work in the src directory. There, all source data are stored, which are then processed on Gulp task on.

In order now to begin the work activate the default task:


gulp

Gulp will activate the server that provides a preview in realtime on the browser - the browser is automatically refreshed after every change on the filesystem. Within the src directory js, Jade and Sass files are monitored for a change in the compiler is active - in the case of Javascript JSHint will check code quality.

I deliberately use no CoffeeScript because it has just performed in teamwork to confusion. If you still want to use CoffeeScript, only the corresponding plugin must install.

Those who want to extend the file monitoring to images, can use the extended default task:


gulp extended

A modification of files, as well as adding or deleting will activate the task, eg is from SVG files automatically generates a PNG fallback.

It may be that the Extend task is not responding at the beginning of changes in the image (src/stash/images/) directory. Since no images at the beginning exist that can Gulp watch. Preparing in this case the images on gulp grunt-rebuild-images and then activate the extended task.

Tasks


gulp init

For the first initialization of the project.


gulp

Default Task


gulp extended

Extended default task - monitors image data


gulp rebuild-js

All the Javascript files are rebuilded (combined, copied, checked)


gulp publish

The project data is prepared for production. CSS, JS and image data are compressed. The documenation with sassdoc is generated.


gulp jade-rebuild

Rebuild all Jade Files


gulp bump

Patch the Version Number from package.json and bower.json (0.0.1)


gulp bump-minor

Extend the Minor Version Number from package.json and bower.json (0.1.0)


gulp bump-major

Extend to the next Release Number from package.json and bower.json (1.0.0)


gulp grunt-rebuild-images

Rebuild all Image Files.


gulp grunt-responsive-images

Build different Sizes from HTML Images

Structure

As already explained, one will usually always work in `src` directory.

root
.. dist/ distribution
.. src/ source

The internal structure of the dist / directory looks like this:

dist/
.. assets/
.... css/ CSS files
.... img/ CSS included images
...... system/ Systemimages like favicon
...... single/ CSS single images
...... textures/ CSS texture images
.... js/ JS files
.... fonts/ Webfonts & iconfonts
.. images/ HTML included images

If you therefore should not be satisfied, you can change the structure of the package.json:

If the paths are changed, the paths must be in the Sass setup file (src/sass/_setup.scss -> $kittn-directorys) also be changed accordingly

This should happen before the gulp init of course, otherwise you have to rename the directories manually, or you delete dist simple and then starts the gulp init again.

The internal structure of the src / directory:

src/
.. jade/ Jade files
.. js/ JS files
.. markdown/ External texts in Markdown format
.. sass/ Sass framework
.. stash/ filing
.... .sytem/ Internal system files
.... bower/ Bower dependencies
.... fonts/ Font files
.... images/ Image sources
...... css-images/ CSS included images
........ single-assets/ Place for each single assets
........ sprite-assets/ Place for each sprite assets
........ svg-assets/ Place for each svg assets with a generated png fallback
........ svgonly-assets/ Place for each svg assets without fallbacks
........ texture-assets/ Place for each texture asset
...... html-images/
........ html-assets/ HTML included images
.... scripts/ Script files that are not installed on Bower

Build HTML with Jade

Jade is a HTML compiler that is in principle used for the first generation of the HTML files, then it is the developer free to decide for themselves whether they want to use Jade on. For example, who works directly with WordPress will not need Jade.

The Jade Files can also be found under src/:

src/
.. jade/ Jade files
.... components/ Jade componenten
.... mixins/ Jade Mixins
.... templatepart/ Template segments, eg HTML Header
.... templates/ Different template types

The Jade part is more like a little boilerplate. It's up to you how much you want to segment the Jade code.

Working with JS Files

Internal Files

Internal JS files are in the directory src/js/ saved, there will be discover app.js file - regularly it can be considered sufficient one file.

The app.js will load in HTML always at the end.

Once the default Gulp task is active, all files will be monitored in src/js/, at a change through this one JSHint test and are stored in dist/assets/js/ (with the same filename).

External Files

Let us turn to the external script files, this can be either Bower load in the package, if they are not found there, one can integrate over the scripts folder.

If you want to, for example, Add an additional scripts, you will first search for it - sometimes the packages are stored under a different name.


bower search package-name

If you have the appropriate package found, it is installed with the following command:


bower install package-name --save

Das entsprechende Paket wird in src/stash/bower/ installiert und zusätzlich in der bower.json notiert. Möchte man ein installiertes Package wieder deinstallieren:

The appropriate package is installed in src/stash/bower/ and also listed in the bower.json. If you want to uninstall an installed package again:


bower uninstall package-name --save

The contents of the bower Folders is not transferred to the external Git repo. Developers who work on the project, download the Dependencies Bower over bower install. Therefore, it is important to note each dependency in the bower.json.

Of course it can happen that you can not find certain packages via Bower, you can register then either yourself:


bower register https://url.to-the.git.repo package-name

Or you can load the package in src/stash/scripts/ where it is available for further use.

To use JS files, they must first be processed with Gulp. JS files are then either in the JS directory dist/assets/js/ copied or they are combined and copied to the JS directory.

Extend Kittn

Kittn is designed to give the developer a skeleton in his hand, with which he can already work. Nevertheless, it will happen that not enough a the default plugins. One or the other would take the native js might use CoffeeScript might be missing that some other feature.

To expand Kittn you should list the new plugins on the package.json, as the node_modules directory is not transmitted to the git repo.


npm install packagename --save-dev

This is a package about NPM Network, installed into the project directory and listed in the package.json. Thus, the subsequent Developer installed exactly the packages that are needed for the project. It understood then you then the gulpfile.js and / or gruntfile.js must modify file.

To uninstall packages:


npm uninstall packagename --save-dev
